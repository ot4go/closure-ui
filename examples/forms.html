Form components

Self-contained demo of the form components in the closure-ui library. Everything works without a server (opening the file via file://): the page loads the bundle with a plain <script> tag and makes no network requests.

This page is deliberately pure HTML — it has no stylesheet at all, so everything you see is the components' built-in default styling. For restyling via CSS variables see theming.html.

1. <closure-form-row> + <closure-form-field>

<closure-form-row> lays out its child fields (<closure-form-field>) in a CSS grid when cols is given, or with flexbox when it is not. Each field contributes a label, a required indicator and validation messages.

Grid with cols, gap and a required field

cols="*,4em,6em": * becomes 1fr, integers become Nfr and anything else passes verbatim to the grid template. required adds the asterisk to the label.

<closure-form-row cols="*,4em,6em" gap="14px">
  <closure-form-field label="First name" required>
    <input type="text" name="fname">
  </closure-form-field>
  <closure-form-field label="MI">
    <input type="text" name="mi" maxlength="1">
  </closure-form-field>
  <closure-form-field label="Date of birth">
    <input type="date" name="dob">
  </closure-form-field>
</closure-form-row>

Label variants: labels="top | side | right | checkbox-right"

The row's labels attribute sets the position of every label; each field can override it with its own labels attribute. The checkbox-* variants are meant for checkboxes.

<closure-form-row cols="*,*">
  <closure-form-field label="Label on top (top, default)">
    <input type="text">
  </closure-form-field>
  <closure-form-field label="On the right" labels="right">
    <input type="text">
  </closure-form-field>
</closure-form-row>

<closure-form-row labels="side">
  <closure-form-field label="Side label" flex="1">
    <input type="text">
  </closure-form-field>
</closure-form-row>

<closure-form-row labels="checkbox-right">
  <closure-form-field label="I accept the terms">
    <input type="checkbox" name="terms">
  </closure-form-field>
</closure-form-row>

warning and error states (live toggling)

Setting or removing the warning="..." / error="..." attributes on the field re-renders only the message. If both are present, error wins over warning.

<closure-form-field id="state-field" label="New password">
  <input type="text">
</closure-form-field>

<script>
  var f = document.getElementById('state-field');
  // warning: amber border + message
  f.setAttribute('warning', 'Must be at least 12 characters');
  // error: red border + message (takes priority over warning)
  f.setAttribute('error', 'The password is required');
  // clearing both hides the message
  f.removeAttribute('warning'); f.removeAttribute('error');
</script>

Readonly mode

The row defines the --cfr-ro-bg, --cfr-ro-color, --cfr-ro-border and --cfr-ro-label tokens that style fields carrying the cfr-ro class (the class that <closure-template> applies automatically in its readonly edit mode; here we add it by hand together with the input's native readonly).

<closure-form-row cols="*,*">
  <closure-form-field class="cfr-ro" label="User">
    <input type="text" value="pbotella" readonly>
  </closure-form-field>
  <closure-form-field class="cfr-ro" label="Department">
    <input type="text" value="Systems" readonly>
  </closure-form-field>
</closure-form-row>

Responsive collapse with min and hide-on-collapse

With min="600px" the row installs a ResizeObserver: below that width the cfr-collapsed attribute kicks in, the grid collapses to a single column, and fields with hide-on-collapse disappear.

Resize the browser window (below ~600 px of row width) to see the collapse. The “Internal notes” field is hidden in collapsed mode.

<closure-form-row cols="*,*,*" min="600px" gap="12px">
  <closure-form-field label="Street" required>
    <input type="text" name="street">
  </closure-form-field>
  <closure-form-field label="City">
    <input type="text" name="city">
  </closure-form-field>
  <closure-form-field label="Internal notes" hide-on-collapse>
    <input type="text" name="notes">
  </closure-form-field>
</closure-form-row>

2. <credential-pwd>

Masked password field that wraps a hidden <input type="password"> (so the value participates in form submission) while showing bullet glyphs ● to the user. Designed to defeat stored-credential autofill on shared admin screens.

Plain, required, and existing value (has-value)

has-value preloads placeholder bullets indicating a password is already on file: the first focus wipes them and starts a fresh input (there is no edit-in-place; the whole new password must be typed).

Paste: on paste, the pasted string replaces the entire value and pasted = true is exposed. A single Backspace clears all of the pasted value at once, and typing after a paste also discards it: there is no character-by-character editing of pasted content.

<credential-pwd name="pwd1"></credential-pwd>
<credential-pwd name="pwd2" required></credential-pwd>
<credential-pwd name="pwd3" has-value></credential-pwd>
<credential-pwd name="pwd4" readonly has-value></credential-pwd>

Enter activates an external button: enter-btn-id

In dialogs where the action button lives outside the form, enter-btn-id points at the element to activate when Enter is pressed. Type something and press Enter (or the button): the component's .value property is read.

(no value yet)
<credential-pwd id="dialog-pwd" name="op-pwd"
                enter-btn-id="btn-accept"></credential-pwd>

<!-- button outside the form -->
<button id="btn-accept" type="button">Accept</button>

<script>
  document.getElementById('btn-accept').addEventListener('click', function() {
    var pwd = document.getElementById('dialog-pwd');
    out.textContent = 'Typed value: ' + pwd.value
                    + '  (pasted: ' + pwd.pasted + ')';
  });
</script>

Enter submits the enclosing form

Without enter-btn-id, Enter triggers the enclosing <form>'s submit. Here we intercept the submit with preventDefault() and show the value, so the demo works without a server.

(the form has not been submitted yet)
<form id="login-form">
  <credential-pwd id="form-pwd" name="password" required></credential-pwd>
  <button type="submit">Sign in</button>
</form>

<script>
  document.getElementById('login-form').addEventListener('submit', function(e) {
    e.preventDefault(); // offline demo: don't navigate
    out.textContent = 'submit intercepted — password = '
                    + document.getElementById('form-pwd').value;
  });
</script>

3. <closure-checkbox-tree>

Cascading checkbox tree. The structure is declared in light DOM with <cbt-item> (which never renders itself); the actual checkboxes live in the Shadow DOM. It is form-associated: it submits a single field whose value is JSON [[path, v, vt], …] where v is the leaf state (0/1, null on parents) and vt the subtree state (0=none, 1=all, 2=custom, null on leaves). Parents have tri-state semantics (indeterminate when mixed).

Expanded mode (expanded) + initial values via src

Initial values are seeded from a JSON data island (<script type="application/json">) referenced by id in the src attribute. Important: paths include the tree name and have no trailing slash: ["/privileges/users/view", 1, null]. Entries whose path does not exist in the tree are ignored. The output below shows the form value (.value) and updates on every change event (which bubbles).

(loading initial value…)
<script type="application/json" id="privileges-data">
  [
    ["/privileges/users/view", 1, null],
    ["/privileges/users/edit", 1, null],
    ["/privileges/reports/export", 1, null]
  ]
</script>

<closure-checkbox-tree name="privileges" expanded src="privileges-data">
  <cbt-item name="users" label="Users">
    <cbt-item name="view"   label="View"></cbt-item>
    <cbt-item name="edit"   label="Edit"></cbt-item>
    <cbt-item name="delete" label="Delete" tip="Permanent action"></cbt-item>
  </cbt-item>
  <cbt-item name="reports" label="Reports">
    <cbt-item name="view"   label="View"></cbt-item>
    <cbt-item name="export" label="Export"></cbt-item>
  </cbt-item>
</closure-checkbox-tree>

<script>
  tree.addEventListener('change', function() {
    out.textContent = tree.value;   // JSON [[path, v, vt], ...]
  });
</script>

Collapsed mode (default): All / None / Custom buttons

Without expanded, the tree renders as a pill with three buttons, labelled All / None / Custom by default (customisable via label-all, label-none and label-custom). The “Custom” button opens a <closure-lightbox> with the full tree (works offline; picks commit on close). The value format is identical to expanded mode.

(no changes yet)
<closure-checkbox-tree name="permissions"
                       label-all="All" label-none="None"
                       label-custom="Custom">
  <cbt-item name="read"  label="Read"></cbt-item>
  <cbt-item name="write" label="Write"></cbt-item>
  <cbt-item name="admin" label="Administration">
    <cbt-item name="users"   label="Users"></cbt-item>
    <cbt-item name="backups" label="Backups"></cbt-item>
  </cbt-item>
</closure-checkbox-tree>

Read-only (readonly)

readonly disables every checkbox in the Shadow DOM (and the collapsed-mode buttons).

<closure-checkbox-tree name="audit" expanded readonly src="ro-data">
  <cbt-item name="access"  label="Access log"></cbt-item>
  <cbt-item name="changes" label="Change log"></cbt-item>
</closure-checkbox-tree>

4. <closure-checkbox-group> + <closure-summary>

The group bundles several trees into a single form field (the child trees no longer submit individually). The output attribute picks the serialisation:

  • output="flat" (default): a single array [[path, v, vt], …] with every tree's entries concatenated in DOM order; each path starts with /<treeName>/…, so the server can demultiplex.
  • output="sections": an object { "tree1": [...], "tree2": [...] } with one key per tree.

Additionally, summary="id" pairs the group with a <closure-summary> (which points back via source="id"): the summary refreshes by itself on every change.

Paired summary (refreshes live)

(tick a checkbox to see the group value)
<closure-checkbox-group id="privs-group" name="privileges"
                        output="sections" summary="privs-summary">
  <closure-checkbox-tree name="users" expanded>
    <cbt-item name="view" label="View users"></cbt-item>
    <cbt-item name="edit" label="Edit users"></cbt-item>
  </closure-checkbox-tree>
  <closure-checkbox-tree name="reports" expanded>
    <cbt-item name="view"   label="View reports"></cbt-item>
    <cbt-item name="export" label="Export reports"></cbt-item>
  </closure-checkbox-tree>
</closure-checkbox-group>

<closure-summary id="privs-summary" source="privs-group"></closure-summary>

<!-- output="flat"     -> [["/users/view",1,null], ["/reports/view",0,null], ...]
     output="sections" -> { "users": [...], "reports": [...] } -->

5. <fingerprint-hands>

Two-hand SVG diagram for capturing or displaying which fingerprints have been recorded. Form-associated. Fingers: l1…l5 (left hand, l1 = thumb) and r1…r5 (right hand). Per-finger states: 1/on = captured (green), 0/off = empty (grey), -1/disabled = palm-coloured and ignored by the form.

Keyboard: the component gets tabindex="0" automatically; the arrow keys walk the fingers in anatomical order (left pinky → left thumb → right thumb → right pinky), wrapping at both ends and skipping disabled fingers. Space or Enter act on the focused finger. The focus ring only appears for keyboard navigation (:focus-visible).

Default, preset fingers, disabled finger and toggle

This example is interactive (toggle): click / Space / Enter flip the finger. Preset: left thumb (l1="1") and right index finger (r2="on"); the left ring finger is disabled (l4="disabled"). The output shows .value and updates on the finger-click event.

(loading…)
<fingerprint-hands name="fingerprints" size="lg" toggle
                   l1="1" r2="on" l4="disabled"></fingerprint-hands>

<script>
  hands.addEventListener('finger-click', function(e) {
    // e.detail = { finger: 'r2', state: 'on' }  (state after the click)
    out.textContent = hands.value;
    // l1:1,l2:0,l3:0,l4:-1,l5:0,r1:0,r2:1,r3:0,r4:0,r5:0
  });
</script>

Sizes (size="sm | md | lg | xl | xxl")

Widths of 80 / 120 / 160 / 220 / 300 px. Default is md.

<fingerprint-hands size="sm" l1="1"></fingerprint-hands>
<fingerprint-hands size="md" l1="1" r1="1"></fingerprint-hands>
<fingerprint-hands size="lg" l1="1" r1="1" r2="1"></fingerprint-hands>

Read-only (readonly) and bulk-set via value

With readonly the fingers render but are not interactive. The value attribute accepts the form-value format to set every finger at once. Without toggle, clicks still fire finger-click but do not change state (useful when the host wants to react to clicks externally).

<fingerprint-hands readonly size="md"
    value="l1:1,l2:1,l3:0,l4:0,l5:0,r1:1,r2:0,r3:0,r4:-1,r5:0">
</fingerprint-hands>
